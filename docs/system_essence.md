# Информационно-аналитическая система сравнительного анализа объектов

## 1. Что это за система

Система предназначена для сравнительного анализа объектов, представленных в табличном виде. В качестве объектов могут выступать квартиры, автомобили, земельные участки, коммерческие предложения, товары, поставщики, компании, инвестиционные проекты и любые другие сущности, которые можно описать набором признаков.

Главная идея системы состоит в том, чтобы превратить произвольный пользовательский датасет в управляемую аналитическую модель, где:

- пользователь сам определяет, какие признаки участвуют в анализе;
- для каждого признака можно задать смысл, тип, способ обработки и правила сравнения;
- итоговый результат получается не как «черный ящик», а как объяснимый рейтинг или список аналогов;
- вся цепочка от загрузки файла до расчета результата воспроизводима и пригодна для повторного запуска.

Система реализована как web-приложение с клиент-серверной архитектурой. Пользователь работает через браузерный интерфейс, а все операции по обработке данных, аналитике, хранению истории и подготовке отчетов выполняются на стороне сервера.

## 2. Какую задачу решает система

В реальных прикладных сценариях сравнение объектов редко сводится к одному параметру. Обычно приходится одновременно учитывать несколько разнородных признаков:

- числовые показатели;
- категории;
- бинарные признаки;
- даты;
- географические координаты;
- признаки с единицами измерения;
- признаки со смешанным качеством данных, пропусками и выбросами.

Проблема состоит в том, что исходные пользовательские данные почти никогда не бывают готовы к сравнению сразу. В них могут встречаться:

- пропуски;
- разные форматы записи;
- числовые значения в виде строк;
- единицы измерения внутри ячеек;
- разные масштабы признаков;
- категориальные поля, требующие ручного кодирования;
- уникальные идентификаторы, которые нельзя использовать как полноценные критерии анализа.

Система решает эту задачу поэтапно:

1. Загружает и распознает датасет.
2. Формирует его профиль и предварительное описание.
3. Позволяет пользователю настроить подготовку данных.
4. Позволяет задать критерии сравнения и веса.
5. Выполняет либо ранжирование, либо поиск аналогов относительно целевого объекта.
6. Формирует объяснимый аналитический результат.
7. Сохраняет историю запуска и позволяет повторно использовать сценарий.

## 3. Ключевая концепция системы

Система не просто «считает score». Она строит управляемый аналитический конвейер, в котором у каждого этапа есть собственная роль:

- импорт преобразует внешний файл во внутреннее представление;
- профилирование помогает понять структуру и качество данных;
- предобработка делает признаки пригодными для сравнения;
- аналитическое ядро рассчитывает близость или итоговую оценку;
- слой результатов объясняет, почему объект оказался выше или ниже других;
- слой истории и отчетности делает анализ воспроизводимым.

Таким образом, система совмещает три уровня:

- подготовку данных;
- сравнительный анализ;
- прикладное объяснение результата.

## 4. Основные пользовательские сценарии

### 4.1 Ранжирование объектов

Пользователь загружает датасет, выбирает признаки, настраивает веса критериев и получает итоговый рейтинг объектов. Такой режим подходит, когда нужно выбрать лучший объект из множества альтернатив.

Примеры:

- выбор лучшей квартиры;
- выбор наиболее подходящего автомобиля;
- ранжирование поставщиков;
- ранжирование инвестиционных предложений.

### 4.2 Поиск аналогов

Пользователь выбирает целевой объект и настраивает критерии, по которым система ищет наиболее близкие объекты. Такой режим подходит, когда нужно не найти «лучший» объект, а подобрать сопоставимые аналоги.

Примеры:

- подбор аналогичных объектов недвижимости;
- поиск похожих автомобилей;
- поиск похожих товаров;
- поиск сопоставимых коммерческих предложений.

### 4.3 Оценка рыночной стоимости по аналогам

Если пользователь работает в режиме поиска аналогов и дополнительно включает оценку стоимости, система использует найденные аналоги для расчета ориентировочной цены целевого объекта.

При этом:

- пользователь выбирает колонку стоимости;
- это поле исключается из расчета близости;
- система оценивает стоимость по аналогам, а не подгоняет аналоги по цене;
- результат показывается как отдельный аналитический блок.

## 5. Архитектура системы

Система построена по клиент-серверной модели и состоит из четырех основных слоев:

### 5.1 Frontend

Frontend реализован на React и TypeScript. Он отвечает за:

- взаимодействие с пользователем;
- пошаговый workflow работы с датасетом;
- настройку параметров анализа;
- визуализацию результатов;
- экспорт клиентских отчетов;
- работу с историей и административной панелью.

### 5.2 Backend

Backend реализован на FastAPI и Python. Он отвечает за:

- прием и обработку пользовательских файлов;
- профилирование датасетов;
- предобработку признаков;
- выполнение аналитических расчетов;
- формирование истории и системной статистики;
- выдачу данных через REST API.

### 5.3 Реляционное хранилище

PostgreSQL используется для хранения структурированных сущностей системы:

- пользователей;
- ролей;
- проектов;
- истории расчетов;
- метаданных файлов;
- параметров запуска;
- системной статистики.

### 5.4 Объектное хранилище

MinIO, совместимое с S3, используется для хранения:

- исходных датасетов;
- сформированных файлов результатов;
- экспортируемых отчетов;
- связанных файловых артефактов.

## 6. Внутренняя структура backend

Серверная часть организована как единое приложение с выделенными функциональными сервисами.

### 6.1 `import_parser.py`

Отвечает за импорт пользовательских файлов в форматах CSV, XLSX и JSON. После чтения любой входной формат приводится к единому табличному представлению на базе `pandas.DataFrame`.

На этом этапе выполняются:

- определение формата файла;
- чтение содержимого;
- нормализация колонок;
- формирование preview;
- определение базовых характеристик полей;
- подготовка внутреннего представления строк.

### 6.2 `profiling_engine.py`

Строит профиль датасета и профили отдельных полей. Профилирование используется как для первичного понимания данных, так и для построения интерфейса подготовки.

Профиль включает:

- предполагаемый тип поля;
- количество пропусков;
- число уникальных значений;
- примеры значений;
- статистику по числовым полям;
- распределения;
- потенциальные выбросы;
- информацию о единицах измерения;
- рекомендации по предобработке.

### 6.3 `measurement_parsing.py`

Отдельный сервис для разбора значений с единицами измерения. Он нужен для случаев, когда значения записаны не как чистые числа, а как строки вида:

- `102,300 km`;
- `85 mi`;
- `57 kg`;
- `1200 g`;
- `₹ 1,25,000`.

Сервис позволяет:

- извлекать числовую часть значения;
- определять единицы измерения;
- приводить значения к целевой единице;
- поддерживать смешанные единицы внутри одного столбца.

### 6.4 `preprocessing_engine.py`

Сердце этапа подготовки данных. Именно этот модуль превращает исходный датасет в аналитически пригодный набор признаков.

### 6.5 `analysis_engine.py`

Выполняет сравнительный анализ объектов. Поддерживает:

- режим ранжирования;
- режим поиска аналогов;
- расчет вкладов критериев;
- чувствительность;
- устойчивость рейтинга;
- пояснение результата.

### 6.6 `pipeline_engine.py`

Оркестратор общего процесса. Он связывает импорт, профилирование, предобработку, анализ, сохранение истории и дополнительные функции вроде оценки стоимости по аналогам.

### 6.7 Прочие backend-сервисы

В проекте также используются:

- `file_service.py` для работы с файлами и историей;
- `user_service.py` для управления пользователями и административной статистикой;
- `security.py` для аутентификации и JWT;
- `object_service.py` для объектных сущностей;
- `dataset_artifact_service.py` и `profile_artifact_service.py` как инфраструктурные слои, которые могут использоваться для повторного доступа к данным и профилям.

## 7. Внутренняя структура frontend

Основная пользовательская логика сосредоточена в:

- `frontend/src/App.tsx`;
- `frontend/src/api.ts`;
- `frontend/src/types.ts`;
- `frontend/src/styles.css`.

Отдельным переиспользуемым компонентом выступает:

- `frontend/src/components/TargetObjectPreviewCard.tsx`.

Frontend организован как пошаговый интерфейс с несколькими стадиями анализа.

## 8. Пользовательский workflow

### 8.1 Этап 1. Загрузка и паспорт данных

На первом этапе пользователь:

- загружает файл;
- получает предпросмотр исходного датасета;
- видит паспорт данных;
- может оценить структуру колонок, количество строк, пропуски и уникальные значения.

Важно, что на первом этапе показывается именно исходный, а не преобразованный датасет.

### 8.2 Этап 2. Подготовка данных

Это самый насыщенный этап. Здесь пользователь не просто подтверждает импорт, а вручную управляет аналитическим смыслом признаков.

Подготовка разбита на несколько вкладок:

- предпросмотр;
- типы данных;
- энкодинг;
- пропуски;
- выбросы;
- масштабирование.

### 8.3 Этап 3. Критерии

Пользователь:

- выбирает режим анализа;
- задает критерии;
- указывает направления сравнения;
- задает веса;
- в режиме аналогов выбирает целевой объект;
- при необходимости задает радиус поиска;
- при необходимости включает оценку стоимости по аналогам.

### 8.4 Этап 4. Результаты

На этапе результатов пользователь получает:

- рейтинг объектов;
- визуализацию score;
- пояснение по выбранному объекту;
- таблицу вкладов критериев;
- графики сравнения;
- аналитику устойчивости;
- при необходимости блок оценки стоимости.

## 9. Логика импорта данных

Система поддерживает форматы:

- CSV;
- XLSX;
- JSON.

Импорт не ограничивается простым чтением файла. После загрузки система:

- нормализует названия колонок;
- формирует preview;
- выделяет признаки;
- строит начальный профиль полей;
- определяет базовые рекомендации по типам.

Таким образом, импорт выступает не как одно действие чтения, а как начало аналитического конвейера.

## 10. Логика профилирования

Профилирование отвечает на вопрос: что собой представляет загруженный датасет до начала ручной настройки.

Система определяет:

- общее количество строк;
- перечень полей;
- типы данных;
- долю и распределение пропусков;
- статистику по числовым признакам;
- набор категорий;
- возможные выбросы;
- возможные единицы измерения;
- рекомендации по предобработке.

Профиль используется:

- в паспорте данных;
- в метриках качества;
- в подсказках пользователю;
- в графиках на этапе подготовки.

## 11. Логика предобработки

Предобработка в системе принципиально управляемая. Пользователь не просто подтверждает автоматически выбранный сценарий, а может вручную менять практически все важные параметры.

### 11.1 Типы данных

Система различает:

- `text`;
- `categorical`;
- `binary`;
- `integer`;
- `float`;
- `datetime`;
- `geo_latitude`;
- `geo_longitude`.

Для дробных полей можно указывать точность. Для дат можно задавать формат. Для координат можно явно назначать роль поля.

### 11.2 Исключение признаков

Пользователь может отключить поля от дальнейшего анализа. Это важно, потому что не все колонки полезны как признаки сравнения.

Например, часто исключаются:

- внутренние идентификаторы;
- уникальные названия;
- вспомогательные текстовые поля;
- служебные колонки.

### 11.3 Поля, используемые в названии объекта

Система поддерживает отдельную настройку `use_in_label`. Она позволяет указать одно или несколько полей, которые используются для построения человекочитаемого названия объекта.

Это важно, потому что:

- целевой объект на этапе аналогов должен выбираться не по голому `id`;
- на этапе результатов объект должен отображаться с понятным названием;
- название объекта не обязано участвовать в самом сравнении.

### 11.4 Работа с пропусками

Для каждого поля пользователь может выбрать стратегию обработки пропусков. В зависимости от типа поля применяются разные подходы:

- удаление строк;
- замена средним;
- замена медианой;
- замена модой;
- замена константой;
- сохранение как есть.

На этапе интерфейса дополнительно доступны:

- preview строк с пропусками;
- матрица пропусков;
- агрегированные показатели качества данных.

### 11.5 Работа с выбросами

Для числовых полей система поддерживает анализ выбросов по IQR и связанный визуальный слой:

- гистограммы;
- ящик с усами;
- количество найденных выбросов;
- настройка порога;
- выбор действия.

Пользователь может:

- не применять обработку;
- удалять выбросы;
- обрезать значения;
- настраивать чувствительность через порог.

### 11.6 Энкодинг категорий

Для категориальных полей в системе используется ручная порядковая шкала. Это сделано осознанно, потому что в прикладном сравнительном анализе one-hot не всегда дает интерпретируемый результат и часто неудобен для ручного задания значимости.

Пользователь видит:

- список категорий;
- частоты встречаемости;
- рекомендуемую ручную шкалу;
- возможность изменить числовое представление каждой категории.

### 11.7 Масштабирование

Для числовых признаков поддерживается масштабирование. Оно отделено по смыслу от остальных преобразований, потому что:

- до масштабирования полезно анализировать сырые распределения;
- после масштабирования удобнее считать сравнение и близость;
- пользователь может переключаться между представлением «до» и «после» нормализации в интерфейсе.

### 11.8 Работа с единицами измерения

Если числовое поле содержит значения с единицами измерения, система может:

- распознать единицу;
- определить семейство единиц;
- привести значения к единому виду;
- показать пользователю, какие единицы были найдены;
- дать выбрать целевую единицу.

Это особенно важно для признаков вроде пробега, веса, расстояния и других физических величин.

## 12. Два уровня представления данных

В системе важно различать несколько уровней значений одного и того же признака:

- исходное значение;
- преобразованное значение после приведения типов, единиц, кодирования и других операций;
- нормализованное значение после масштабирования.

Это различие особенно важно:

- на этапе подготовки;
- в карточке целевого объекта;
- в таблице вкладов на этапе результатов.

Благодаря этому пользователь видит не только конечный score, но и путь, который прошли данные до него.

## 13. Логика сравнительного анализа

### 13.1 Режим ранжирования

В режиме ранжирования система рассчитывает итоговую оценку объекта как взвешенную сумму подготовленных значений критериев.

Идея простая:

- каждый критерий имеет вес;
- для каждого объекта берется значение критерия;
- значение умножается на вес;
- сумма вкладов дает итоговый score.

Этот режим отвечает на вопрос: какой объект лучше по выбранной системе критериев.

### 13.2 Режим поиска аналогов

В режиме аналогов логика другая. Здесь система не ищет лучший объект, а сравнивает все объекты с целевым.

Сначала считается отклонение по каждому критерию:

`|x_ij - x_tj|`

Затем отклонения агрегируются с учетом весов. В текущей реализации используется взвешенное среднее отклонение, после чего оно преобразуется в similarity:

`similarity = 1 / (1 + distance)`

Именно поэтому:

- меньшее отклонение дает большую близость;
- большее сходство означает более подходящий аналог.

### 13.3 Географическое ограничение

Если в датасете выделены поля широты и долготы, система может ограничивать поиск аналогов по радиусу относительно целевого объекта.

Это позволяет:

- искать только пространственно близкие объекты;
- использовать географию как реальное ограничение, а не как обычный float-признак;
- улучшать прикладную адекватность аналогов.

## 14. Оценка рыночной стоимости по аналогам

Это специальный режим внутри `analog_search`.

Если пользователь включает оценку стоимости, система:

1. Берет найденные аналоги.
2. Исключает сам целевой объект.
3. Берет только аналоги, у которых есть цена.
4. Выбирает указанное число аналогов.
5. Считает ориентировочную цену целевого объекта.

Важно:

- поле цены исключается из similarity, чтобы система не подбирала аналоги по самой цене;
- цена используется только как поле оценки;
- для оценки берутся значения до масштабирования;
- если similarity положительны, цена считается как средневзвешенная по similarity;
- если similarity вырождаются в ноль, система использует обычное среднее.

Таким образом, оценка стоимости выступает как дополнительный прикладной слой поверх уже найденных аналогов.

## 15. Объяснимость результата

Сильная сторона системы состоит в том, что она не ограничивается только выдачей числа.

Пользователь получает:

- вклад каждого критерия;
- исходные значения;
- преобразованные значения;
- нормализованные значения;
- ранговую позицию объекта;
- графики сравнения;
- пояснения по результату.

На практике это означает, что пользователь может ответить на вопросы:

- почему объект оказался выше другого;
- какие признаки сильнее всего повлияли на результат;
- как изменились данные после подготовки;
- какие аналоги были использованы для оценки стоимости.

## 24. Ограничения системы

При всех своих возможностях система имеет понятные границы применимости:

- качество результата зависит от качества исходного датасета;
- ручной энкодинг категорий требует предметного понимания данных;

- оценка стоимости по аналогам является аналитической рекомендацией, а не юридически значимой оценкой;

Главная ценность системы не в том, что она умеет посчитать один score, а в том, что она объединяет в одном продукте:

- управляемую подготовку данных;
- объяснимый сравнительный анализ;
- поиск аналогов;
- географические ограничения;

- воспроизводимость сценариев;
- историю и административную наблюдаемость.

Иными словами, система превращает произвольный пользовательский датасет в прозрачный прикладной аналитический инструмент, пригодный как для исследования, так и для регулярной практической работы.
