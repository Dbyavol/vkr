8: Abstract
9: The paper considers the development of an information and analytical system for comparative analysis of objects based on tabular data. The system is designed for uploading datasets, their initial profiling and preprocessing, configuring comparison criteria, calculating object rankings, searching for analogues relative to a selected target object, and generating a comparison report.
10: As part of the work, methods of comparative object analysis, existing software solutions, and architectural approaches to building information and analytical systems were analyzed. The developed system is based on a client-server architecture. The server side is implemented in Python using FastAPI, PostgreSQL is used for storing structured data, and the client side is developed as a web interface for step-by-step user interaction with data, criteria, and analysis results.
11: The result of the work is an information and analytical system that can be used for object ranking, analogue search, and decision support in applied domains where alternatives need to be compared by a set of heterogeneous features. Further development may include adding new multi-criteria analysis methods, expanding reporting capabilities, supporting additional data formats, and improving monitoring tools.

44: Задача сравнительного анализа объектов
45: Сравнительный анализ объектов представляет собой процесс сопоставления нескольких альтернатив по набору признаков с целью определения наиболее предпочтительных вариантов, формирования рейтинга или поиска объектов, близких к заданному целевому объекту. Такая задача возникает в различных прикладных областях, где необходимо принять решение на основе совокупности характеристик, а не одного отдельного показателя.
46: Объект сравнения может быть представлен в виде набора параметров. Например, при анализе объектов недвижимости учитываются стоимость, площадь, расположение, транспортная доступность и состояние объекта; при сравнении автомобилей — цена, год выпуска, пробег, расход топлива и технические характеристики; при оценке оборудования — производительность, стоимость, надежность, энергопотребление и условия эксплуатации. Таким образом, каждый объект можно описать вектором признаков, а задача сравнительного анализа заключается в обработке этих признаков и формировании итоговой оценки.
47: В общем виде набор данных для сравнительного анализа может быть представлен в виде таблицы, где строки соответствуют объектам, а столбцы — их характеристикам. При этом признаки могут иметь различную природу: числовую, категориальную или текстовую. Для проведения корректного анализа такие данные необходимо привести к единому виду, пригодному для вычислений. Числовые признаки могут требовать нормализации, категориальные признаки — кодирования, а пропущенные или аномальные значения — предварительной обработки.
48: Особенностью задачи сравнительного анализа является то, что разные признаки могут иметь различное направление предпочтительности. Для одних критериев большее значение является лучшим, например производительность, площадь или рейтинг надежности. Для других критериев предпочтительным является меньшее значение, например стоимость, расход топлива, время ожидания или количество отказов. Поэтому при разработке системы сравнительного анализа необходимо учитывать направление оптимизации каждого критерия.
49: Кроме того, отдельные критерии обычно имеют разную значимость для пользователя. Например, при выборе автомобиля для одного пользователя наиболее важной может быть стоимость, для другого — надежность, а для третьего — расход топлива. Следовательно, система должна предоставлять возможность задавать весовые коэффициенты критериев, отражающие их относительную важность в итоговой оценке. Учет весов позволяет адаптировать результаты анализа под конкретный сценарий использования.
50: Результатом сравнительного анализа может быть ранжированный список объектов, в котором каждому объекту сопоставлена интегральная оценка. Чем выше значение итоговой оценки, тем больше объект соответствует заданным пользователем критериям. Помимо общего рейтинга, важным результатом является объяснение полученной оценки: пользователь должен понимать, какие критерии оказали наибольшее влияние на итоговое положение объекта в рейтинге. Это повышает прозрачность работы системы и снижает субъективность принятия решений.
51: Отдельной задачей является поиск аналогов относительно выбранного целевого объекта. В этом случае система должна определить объекты, наиболее близкие к целевому по выбранным признакам. Такая возможность полезна, когда пользователь хочет найти альтернативы уже известному объекту, например похожие товары, квартиры, автомобили, технические устройства или предложения. Для этого после нормализации признаков может использоваться метрика близости, позволяющая оценить степень сходства объектов.
52: При ручном выполнении сравнительного анализа пользователь часто использует электронные таблицы, самостоятельно задает формулы, нормализует признаки, рассчитывает веса и сортирует результаты. Такой подход может быть удобен для небольших наборов данных, однако при увеличении количества объектов и критериев он становится трудоемким и подверженным ошибкам. Кроме того, ручные расчеты затрудняют повторное использование настроек, сохранение истории расчетов и объяснение полученных результатов.
53: Таким образом, задача сравнительного анализа объектов требует автоматизации нескольких этапов: загрузки и проверки исходных данных, предобработки признаков, выбора критериев, настройки весов, расчета итоговых оценок, поиска аналогов и визуализации результатов. Разрабатываемая информационно-аналитическая система должна обеспечивать выполнение этих этапов в едином пользовательском сценарии, сохраняя при этом интерпретируемость расчетов и возможность адаптации анализа под разные предметные области.

55: В задачах сравнительного анализа объектов широко применяются методы многокритериального анализа, позволяющие оценивать альтернативы на основе набора критериев. Такие методы используются в системах поддержки принятия решений, аналитических платформах и системах оценки объектов. 66: Метод анализа иерархий (AHP)
67: Одним из наиболее распространенных подходов является многокритериальный анализ, который используется для сравнения объектов по нескольким параметрам одновременно. В рамках данного подхода формируется набор критериев, характеризующих объекты, после чего производится их оценка и сопоставление. Многокритериальные методы широко применяются в системах поддержки принятия решений, потому что позволяют учитывать различные аспекты анализируемых объектов и формировать комплексную оценку [7].

80: Метод PAPRIKA
81: Метод PAPRIKA (Potentially All Pairwise RanKings of all possible Alternatives) относится к методам многокритериального анализа и используется для определения относительной важности критериев на основе попарных сравнений альтернатив. Рассматриваемый метод применяется в системах поддержки принятия решений и реализован, например, в аналитической платформе 1000Minds [8].
82: Перед применением метода критерии предварительно приводятся к дискретной шкале: для каждого критерия задается набор уровней, которым затем сопоставляются числовые значения. Для каждого критерия задается набор возможных значений, соответствующих различным уровням качества или предпочтительности. Например, для критерия «стоимость» могут быть выделены уровни «низкая», «средняя» и «высокая». Таким образом, перед применением метода выполняется дискретизация значений критериев, что позволяет формировать альтернативы, состоящие из комбинаций созданных дискретных уровней.

101: Методы машинного обучения
102: В задачах сравнительного анализа объектов могут применяться методы машинного обучения, предназначенные для прогнозирования оценок и ранжирования альтернатив. В отличие от классических методов многокритериального анализа, такие модели строятся на основе обучающей выборки и позволяют автоматически выявлять зависимости между характеристиками объектов и их итоговой оценкой [9].
103: Одним из наиболее распространенных подходов является использование регрессионных моделей, в рамках которых для каждого объекта вычисляется прогнозная оценка на основе его признаков. В простейшем случае такая модель представляется в виде линейной регрессии:

112: Продвинутым вариантом являются модели learning-to-rank, которые используются для построения ранжирования объектов на основе исторических данных. Подобные методы активно применяются в рекомендательных системах, поисковых системах и сервисах подбора аналогов [10]. Их преимуществом является возможность учитывать сложные нелинейные зависимости между признаками объектов и итоговым порядком альтернатив.
113: Вместе с тем применение методов машинного обучения в задачах ранжирования связано с рядом ограничений. Такие модели требуют наличия обучающей выборки, в которой уже известны корректные оценки или порядок объектов. Кроме того, результаты работы модели могут быть недостаточно интерпретируемыми для конечного пользователя. В информационно-аналитических системах поддержки принятия решений такие особенности являются существенным ограничением, потому что пользователю важно понимать, каким образом формируется итоговая оценка и какой вклад в нее вносит каждый критерий.
114: Таким образом, методы машинного обучения могут рассматриваться как перспективный инструмент ранжирования объектов, но для разработки интерпретируемой информационно-аналитической системы целесообразнее является использование метода взвешенных коэффициентов, который обеспечивает прозрачность вычислений и удобство практической реализации.

121: Обзор существующих систем и программных решений
122: В настоящее время задачи сравнительного анализа объектов решаются с использованием различных программных систем и аналитических платформ. Такие системы позволяют сопоставлять альтернативные объекты по набору критериев, определять наиболее предпочтительные варианты и выполнять анализ данных, необходимых для принятия решений.
123: Современные программные средства, используемые для сравнительного анализа объектов, условно разделены на несколько групп: системы многокритериального принятия решений, специализированные системы оценки объектов, а также универсальные инструменты анализа данных. Рассмотрим наиболее распространенные решения, применяемые для сравнительного анализа объектов.
124: Одной из известных систем многокритериального анализа является платформа «1000Minds», предназначенная для поддержки принятия решений на основе метода PAPRIKA, а также для определения предпочтений пользователя и выбора наиболее подходящих альтернатив на основе анализа критериев. В системе реализован интерактивный подход к анализу данных: пользователю предлагается последовательно сравнивать пары альтернатив, после чего система формирует модель предпочтений и рассчитывает веса критериев [11]. Интерфейс системы ориентирован на проведение опросов и экспертных оценок, что делает ее удобной для задач стратегического планирования, управления проектами и оценки различных вариантов решений. Вместе с тем использование системы затруднено при анализе большого количества объектов, потому что процесс сравнения требует значительного числа пользовательских оценок со стороны пользователя. Кроме того, система ориентирована на работу с заранее заданными сценариями анализа и не предоставляет гибких возможностей изменения формулы итоговой оценки альтернатив
125: Другим известным инструментом является система «Expert Choice», которая реализует метод анализа иерархий (AHP), она широко применяется в управлении проектами, стратегическом планировании и инвестиционном анализе. Система «Expert Choice» предоставляет графический интерфейс для построения иерархической структуры критериев и альтернатив, что позволяет формировать модели принятия решений различной сложности. Пользователь задает критерии анализа, формировать иерархию факторов и выполнять сравнительный анализ альтернатив. Система автоматически рассчитывает приоритеты критериев и формирует рейтинг альтернатив. Несмотря на развитые аналитические возможности, использование «Expert Choice» требует выполнения большого числа сравнений между критериями и альтернативами, что усложняет работу при увеличении размеров задачи [12].
126: Похожий подход реализован в системе «Super Decisions», которая также основана на методе анализа иерархий, но использует его расширение – метод аналитических сетей (ANP). В отличие от классического AHP, улучшенный метод позволяет учитывать взаимосвязи и зависимости между критериями и альтернативами [13]. Данная система позволяет проводить сложный многокритериальный анализ, строить модели принятия решений и рассчитывать приоритеты альтернатив. Вместе с тем построение таких моделей требует сложной настройки структуры анализа и глубокого понимания используемых методов, что затруднено для пользователей без специальной подготовки.
127: В некоторых областях применяются специализированные системы анализа объектов, ориентированные на конкретную предметную область. Например, в сфере недвижимости широко используются автоматизированные системы оценки объектов недвижимости (Automated Valuation Models, AVM) [14]. Такие системы предназначены для анализа параметров объектов недвижимости и автоматического расчета их рыночной стоимости. В процессе анализа учитываются различные характеристики объектов, включая площадь, расположение, тип здания, инфраструктуру и динамику цен на рынке. Подобные системы часто используются на платформах недвижимости и в банковских организациях для предварительной оценки стоимости объектов. Основным преимуществом таких решений является высокая степень автоматизации анализа, но данные системы ориентированы на узкую предметную область и не могут использоваться как универсальные инструменты сравнительного анализа различных типов объектов.
128: Помимо специализированных систем, задачи сравнительного анализа часто решаются с использованием универсальных инструментов анализа данных, которые позволяют выполнять расчеты, формировать сравнительные таблицы и рассчитывать итоговые оценки объектов на основе заданных критериев. Наиболее распространенным примером является использование электронных таблиц, таких как Microsoft Excel или Google Sheets. Благодаря доступности и простоте использования электронные таблицы остаются одним из наиболее распространенных инструментов анализа данных, но при работе с большим количеством параметров и объектов такие инструменты становятся неэффективными и требуют значительных временных затрат на подготовку данных и выполнение расчетов [15].
129: Таким образом, анализ существующих программных решений показывает, что большинство систем сравнительного анализа либо ориентировано на узкоспециализированные задачи, либо требует сложной процедуры настройки и проведения большого количества сравнений. Универсальные инструменты анализа данных, в свою очередь, не обладают встроенными механизмами сравнительного анализа объектов и требуют ручной реализации алгоритмов оценки. 141: Монолитная архитектура
142: Монолитная архитектура является одним из наиболее традиционных подходов к разработке программных систем. В рамках данного подхода все функциональные компоненты приложения объединяются в единую систему, в которой логика обработки данных, пользовательский интерфейс и работа с базой данных реализуются в одном программном модуле [17].
143: Основным преимуществом монолитной архитектуры является относительная простота разработки и развертывания системы. Все компоненты приложения находятся в одном проекте, что упрощает управление зависимостями и процесс разработки. При увеличении объема функциональности система становится сложно сопровождать, потому что изменение отдельных компонентов требует пересборки и повторного развертывания всего приложения.
144: Кроме того, монолитная архитектура ограничивает возможности масштабирования системы. При увеличении нагрузки масштабировать приходится все приложение целиком, даже если повышенная нагрузка приходится только на отдельные компоненты системы.

148: уровень бизнес-логики,
149: уровень доступа к данным.
150: Такое разделение позволяет изолировать различные компоненты системы и облегчает процесс разработки и сопровождения программного обеспечения [18]. Многослойная архитектура широко применяется в корпоративных информационных системах и веб-приложениях.
151: При увеличении сложности системы многослойная архитектура приводит к появлению жестких зависимостей между компонентами. Кроме того, масштабирование системы часто ограничивается возможностями отдельных уровней архитектуры.
152: Клиент-серверная архитектура
153: Клиент-серверная архитектура представляет собой архитектурный подход к построению программных систем, при котором приложение разделяется на две основные части: клиентскую и серверную. Клиентская часть отвечает за взаимодействие с пользователем и отправку запросов, тогда как серверная часть выполняет обработку данных, реализацию бизнес-логики и управление хранилищем данных [19].
154: В рамках данной архитектуры клиентское приложение отправляет запросы серверу через сетевые протоколы, после чего сервер обрабатывает полученные данные и возвращает результат выполнения запроса. Взаимодействие между клиентом и сервером чаще всего осуществляется через HTTP-протокол и программные интерфейсы (API). Такой подход широко применяется при разработке веб-приложений и информационных систем.
155: Одним из основных преимуществ клиент-серверной архитектуры является разделение ответственности между компонентами системы. Клиентская часть отвечает за отображение данных и пользовательский интерфейс, тогда как серверная часть обеспечивает выполнение вычислений, обработку запросов и работу с базами данных. Такой подход позволяет повысить удобство разработки системы и облегчить ее сопровождение.
156: При высокой нагрузке сервер становится узким местом системы, потому что на нем выполняется основная обработка запросов. Для решения данной проблемы применяются методы масштабирования серверной части и распределения нагрузки.
157: Архитектура микросервисов
158: Одним из современных подходов к разработке программных систем является архитектура микросервисов. В рамках данного подхода система строится как набор независимых сервисов, каждый из которых реализует определенную функцию и развертывается отдельно от других компонентов системы [20].
159: Каждый микросервис представляет собой самостоятельный модуль, имеющий собственную бизнес-логику и, при необходимости, собственное хранилище данных. Взаимодействие между сервисами осуществляется через программные интерфейсы, как правило с использованием REST-API или других сетевых протоколов.
160: Основными преимуществами микросервисной архитектуры являются:
161: высокая масштабируемость системы;
162: возможность независимого развертывания компонентов;
163: удобство сопровождения и развития системы;
164: возможность использования различных технологий для реализации отдельных сервисов.
165: Микросервисная архитектура широко применяется при разработке современных веб-приложений и аналитических платформ, где требуется обработка больших объемов данных и высокая гибкость системы.
166: Использование микросервисной архитектуры также связано с рядом сложностей. В частности, разработка таких систем требует решения задач распределенного взаимодействия сервисов, управления сетевыми вызовами и обеспечения отказоустойчивости системы.
167: Выбор архитектуры для разрабатываемой системы
168: Проведенный анализ архитектурных подходов показывает, что выбор архитектуры системы определяется особенностями решаемой задачи. Для информационно-аналитических систем сравнительного анализа объектов характерны следующие требования: возможность обработки значительного количества данных, гибкость при добавлении новых аналитических модулей и масштабируемость системы при увеличении нагрузки.
169: Монолитная архитектура обладает относительной простотой разработки, но при увеличении функциональности системы возникают сложности с ее сопровождением и масштабированием.
170: Микросервисная архитектура, напротив, обеспечивает независимость компонентов и масштабируемость, однако требует более сложной инфраструктуры, организации сетевого взаимодействия между сервисами и дополнительных средств мониторинга.
171: Многослойная и клиент-серверная архитектуры позволяют разделить систему на логические компоненты, но при увеличении количества функций и сервисов могут возникать сильные зависимости между компонентами системы.
172: С учетом указанных факторов для разработки системы была выбрана клиент-серверная архитектура с выделением отдельных модулей на стороне серверного приложения. Клиентская часть отвечает за взаимодействие с пользователем, загрузку данных, настройку параметров анализа и визуализацию результатов. Серверная часть реализует основную бизнес-логику: профилирование и предобработку данных, сравнительный анализ объектов, хранение истории расчетов, авторизацию и формирование отчетов.
173: Внутри серверного приложения функциональность разделена на отдельные логические модули. Такой подход позволяет сохранить простоту разработки и развертывания, но при этом обеспечивает структурное разделение компонентов системы. В дальнейшем отдельные модули, например модуль предобработки данных или модуль сравнительного анализа, могут быть вынесены в самостоятельные сервисы при необходимости масштабирования.
174: Таким образом, выбранная архитектура обеспечивает баланс между простотой реализации, удобством сопровождения, логическим разделением компонентов и возможностью дальнейшего развития системы.
175: Выбор инструментов и технологий реализации
176: При разработке информационно-аналитических систем важную роль играет выбор технологий и инструментов реализации. От выбранных программных средств зависит производительность системы, удобство разработки, возможность масштабирования и дальнейшего сопровождения программного обеспечения.
177: В качестве основного языка реализации серверной части выбран Python. Согласно статистическим обзорам отрасли, Python используют около 90 % специалистов по анализу данных, что делает его наиболее распространенным языком программирования в данной области (Рисунок 3) [21]. Высокая популярность Python обусловлена наличием развитой экосистемы библиотек для обработки данных, статистического анализа и разработки серверных приложений  [22]. В рамках разрабатываемой системы Python используется для реализации серверной логики, обработки входных наборов данных, предобработки признаков, вычисления итоговых оценок объектов, поиска аналогов и формирования отчетных материалов
179: Рисунок 3 – Статистика использования инструментов в Data Science
180: Для обработки табличных наборов данных используется библиотека Pandas. Данная библиотека предназначена для чтения, преобразования, фильтрации и анализа структурированных данных. В разрабатываемой системе Pandas применяется для загрузки файлов форматов CSV, XLSX и JSON, формирования предпросмотра датасета, анализа структуры полей, определения типов признаков, поиска пропущенных значений и подготовки данных к дальнейшему аналитическому этапу.
181: Для выполнения численных вычислений применяется библиотека NumPy, обеспечивающая эффективную работу с многомерными массивами данных. Использование NumPy позволяет выполнять операции нормализации признаков, расчет интегральных оценок объектов, вычисление вкладов критериев и определение близости между объектами при поиске аналогов. Совместное использование Pandas и NumPy обеспечивает необходимую основу для реализации алгоритмов сравнительного анализа объектов.
182: Для реализации серверной части системы выбран веб-фреймворк FastAPI, предназначенный для создания веб-сервисов и программных интерфейсов приложений. Выбранный фреймворк основан на стандартах Python и обеспечивает высокую скорость обработки запросов благодаря использованию современных механизмов работы с типизированными данными [23]. Одним из преимуществ FastAPI является автоматическая генерация документации API, что значительно упрощает процесс разработки и тестирования системы.
183: Для описания схем входных и выходных данных используется библиотека Pydantic, тесно интегрированная с FastAPI. Она применяется для валидации параметров анализа, конфигураций предобработки, описаний критериев, весов и структур результирующих данных. Использование Pydantic снижает вероятность ошибок при передаче данных между клиентской и серверной частями системы и позволяет формировать понятные сообщения о некорректных параметрах запроса.
184: Клиентская часть системы реализована с использованием React и TypeScript. React выбран как средство построения одностраничного web-приложения на основе компонентного подхода. Такой подход особенно удобен для разрабатываемой системы, поскольку пользовательский интерфейс включает несколько логически связанных этапов: загрузку данных, предпросмотр датасета, подготовку признаков, настройку критериев, запуск расчета и просмотр результатов анализа. Компонентная модель React позволяет организовать интерфейс как совокупность переиспользуемых независимых блоков.
185: Использование TypeScript повышает надежность клиентской части за счет статической типизации. Это особенно важно в условиях интенсивного обмена структурированными данными между клиентом и сервером. Клиентская часть получает от API описания колонок датасета, конфигурации предобработки, критерии анализа, результаты ранжирования объектов, сведения о вкладе отдельных признаков и параметры отчетов. Типизация таких структур уменьшает вероятность ошибок в пользовательском интерфейсе и упрощает поддержку проекта.
186: Для сборки клиентского приложения используется инструмент Vite, обеспечивающий быстрый запуск проекта в режиме разработки и эффективную сборку итоговой версии приложения. Использование Vite упрощает процесс разработки пользовательского интерфейса и ускоряет внесение изменений в компоненты.
187: Для хранения структурированных данных в системе используется PostgreSQL. Данная система управления базами данных является одной из наиболее распространенных реляционных СУБД и отличается высокой производительностью и обеспечением надежного хранения данных [24]. Для взаимодействия серверной части с реляционной базой данных используется SQLAlchemy, обеспечивающая объектно-реляционное отображение и удобную работу с сущностями системы на уровне Python-кода.
188: Для хранения исходных файлов пользователей и сформированных отчетов целесообразно использовать объектное хранилище. В разрабатываемой системе для этой цели применяется S3-совместимое хранилище MinIO. Такой подход позволяет не перегружать реляционную базу крупными файлами и хранить в PostgreSQL только метаданные: имя файла, формат, размер, идентификатор пользователя, идентификатор проекта и ссылку на объект в хранилище. Разделение базы данных и файлового хранилища повышает гибкость системы.
189: Для развертывания и запуска системы используется Docker. Контейнеризация позволяет изолировать программные компоненты, обеспечить воспроизводимость окружения и упростить процесс переноса системы между различными вычислительными средами [25]. В рамках разрабатываемой системы Docker используется для запуска серверной части, клиентского приложения, базы данных PostgreSQL и объектного хранилища MinIO.
190: Для управления многокомпонентной средой используется Docker Compose. С его помощью состав системы описывается в едином конфигурационном файле, что позволяет запускать все основные компоненты одной командой. Такой подход удобен при локальной разработке, тестировании, отладке и демонстрации работоспособности программного комплекса.
191: Таким образом, выбранный набор технологий соответствует требованиям разрабатываемой системы. Python, Pandas и NumPy обеспечивают обработку табличных данных и реализацию алгоритмов сравнительного анализа; FastAPI, Pydantic и SQLAlchemy используются для построения серверной части приложения; React, TypeScript и Vite обеспечивают реализацию современного интерфейса; PostgreSQL применяется для хранения структурированных данных и истории расчетов; MinIO используется для хранения исходных файлов и отчетов; Docker и Docker Compose обеспечивают удобное развертывание системы. Совокупность выбранных инструментов позволяет реализовать расширяемую и удобную для пользователя информационно-аналитическую систему сравнительного анализа объектов.

В ходе анализа задач преддипломной практики и требований к прототипу было установлено, что наиболее целесообразным является использование модульного подхода внутри единого серверного приложения. Такой подход обеспечивает логическое разделение функциональных областей системы на самостоятельные подсистемы и одновременно снижает сложность реализации, развертывания и поддержки приложения.

196: Итоговая архитектура системы включает веб-интерфейс пользователя, серверное приложение, реляционную базу данных и объектное хранилище для хранения файлов от пользователя. Веб-интерфейс обеспечивает полный сценарий работы с системой: загрузку исходных данных, анализ структуры датасета, настройку критериев и весов, выполнение вычислений, визуализацию результатов и формирование отчетной документации.
197: Серверное приложение реализовано как единый модуль с выделением функциональных компонентов, отвечающих за основные задачи системы:
198: Управление пользователями и доступом – регистрация, авторизация и разграничение прав;
199: Загрузка и первичная обработка данных – поддержка различных форматов файлов, нормализация структуры и формирование внутреннего представления данных;
200: Аналитическая обработка – расчет интегральных оценок, поиск аналогов и формирование поясняющих отчетов;

203: Выбор такой архитектуры позволяет компонентам сервера взаимодействовать внутри единого приложения через внутренние вызовы функций, что упрощает передачу данных между модулями, повышает надежность и целостность аналитического процесса. При этом логическая модульность сохраняется, обеспечивая удобство сопровождения и возможность расширения функциональности в будущем. Для разработки модулей использовались язык Python и веб-фреймворк FastAPI, что обеспечило строгую типизацию запросов, автоматическую документацию интерфейсов и упрощение интеграции компонентов [26].
204: Для хранения структурированных данных используется реляционная база PostgreSQL, которая содержит сведения о пользователях, проектах, истории расчетов и метаданных файлов [27]. Для хранения исходных наборов данных и сформированных отчетов применяется S3-совместимое объектное хранилище MinIO [28]. Такое разделение позволяет эффективно работать с большими объемами информации, снижает нагрузку на базу данных и упрощает управление пользовательскими файлами.
205: Выбор архитектуры модульного монолита позволил упростить взаимодействие между компонентами серверной части. Вместо передачи данных между отдельными сервисами по сети в случае микросервисной архитектуры обмен осуществляется через прямые вызовы внутренних модулей и функций. Это уменьшает накладные расходы на выполнение запросов, упрощает отладку и делает аналитический конвейер более целостным. При этом логическое разделение системы на независимые подсистемы сохранено, что обеспечивает удобство дальнейшего сопровождения и возможность расширения функциональности.

219: Модуль предобработки реализует процедуры, необходимые для подготовки набора данных к сравнительному анализу. В рамках его работы выполняются профилирование полей, определение пропусков, обнаружение выбросов, анализ распределений признаков, а также подготовка рекомендаций по преобразованию данных. Для числовых полей предусмотрены механизмы нормализации и обработки аномальных значений, а для категориальных и текстовых признаков поддерживается выбор способов кодирования.
220: Важной особенностью модуля является то, что система не навязывает преобразования автоматически, а предоставляет пользователю рекомендации по проблемным признакам. Такой подход делает процесс подготовки данных более управляемым и интерпретируемым, что особенно важно для аналитических задач, в которых пользователь должен понимать влияние предобработки на итоговый результат.
221: Модуль сравнительного анализа
222: Ключевой вычислительный модуль реализует расчет интегральных оценок, поиск структурно близких аналогов и формирование пояснительных метрик. Входные параметры принимаются строгой схемой валидации, содержащей список активных критериев, направления оптимизации, весовые коэффициенты, идентификатор целевого объекта и флаги дополнительных вычислений.
223: Алгоритм выполнения включает следующие этапы:
224: Загрузка подготовленных данных из хранилища по идентификатору файла и конфигурации предобработки.
225: Нормализация активных критериев согласно выбранным направлениям оптимизации.
226: Применение весовых коэффициентов с автоматической проверкой их суммы; при отклонении от единицы более чем на 0,001 выполняется авто-нормализация.
227: Расчет интегральных оценок и формирование ранжированного списка.
228: Поиск аналогов целевого объекта по метрике близости с фильтрацией по порогу сходства и ограничением выдачи.
229: Генерация объясняющего отчета: вклад каждого критерия в итоговую оценку, анализ устойчивости топ- при варьировании весов, расчет показателя надежности.
230: Результат возвращается в формате, готовом для рендеринга графиков и таблиц на клиентской стороне. Особое внимание уделено обработке пограничных случаев: пустых датасетов, датасетов с одним объектом, признаков с нулевым размахом, некорректных весовых распределений. Во всех случаях система формирует структурированные сообщения об ошибках с указанием конкретного поля и рекомендации по устранению проблемы.
231: Модуль хранения данных и истории расчетов
232: Для хранения прикладных данных используется PostgreSQL, в которой размещаются сведения о пользователях, проектах, истории расчетов и метаданных файлов. Исходные датасеты и сформированные артефакты анализа сохраняются в S3-совместимом объектном хранилище. Такое разделение позволяет эффективно работать как со структурированными данными, так и с пользовательскими файлами, не перегружая реляционную базу крупными бинарными объектами. Общая структура базы данных и ее связь с объектным хранилищем представлена на Рисунок 5.
233: Дополнительно реализован механизм проектов и истории сравнений. Это позволяет пользователю сохранять параметры запуска анализа, возвращаться к ранее выполненным расчетам, повторно использовать датасеты и сравнивать результаты при изменении конфигурации критериев и весов.
235: Рисунок 5 – Структура базы данных
236: Для обеспечения целостности данных реализованы транзакционные операции при сохранении истории, проверки уникальности идентификаторов проектов и автоматическая очистка временных артефактов после завершения сессии. Интеграция с объектным хранилищем выполнена через генерацию временных ссылок доступа, что исключает необходимость постоянного переноса файлов между сервисами и снижает нагрузку на сеть.
237: Модуль авторизации и управления доступом
238: Для обеспечения многопользовательского режима в системе реализован модуль авторизации и управления доступом. Он поддерживает регистрацию пользователей, вход в систему, хранение учетных записей, выдачу токенов доступа и разграничение ролей. Наличие данного модуля позволяет не только ограничивать доступ к отдельным функциям системы, но и связывать проекты, историю расчетов и сформированные отчеты с конкретным пользователем.
239: Реализация авторизации также улучшает эксплуатационные свойства системы, поскольку делает возможным персонализированную работу с данными и контроль пользовательских действий в рамках единой информационно-аналитической среды.
241: Разработка клиентской части и пользовательских сценариев
242: Клиентская часть разработана как одностраничное веб-приложение с использованием современных технологий построения пользовательских интерфейсов. Архитектура клиентской логики исключает выполнение вычислений на стороне пользователя; все аналитические операции делегированы серверной части. Состояние текущего сеанса сохраняется в локальном хранилище браузера, что обеспечивает восстановление параметров при обновлении страницы, закрытии вкладки или потере сетевого соединения.
243: Интерфейс организован в виде последовательного навигатора с пятью этапами: загрузка данных, подготовка, настройка критериев, визуализация результатов и формирование отчета.
244: На первом этапе реализуется загрузка файла и построение предварительной таблицы с фиксированными заголовками и возможностью горизонтальной прокрутки (Рисунок 6).
246: Рисунок 6 – Экран загрузка данных в пользовательском интерфейсе.
247: На этапе подготовки отображаются метрики качества, графики распределений и рекомендации по обработке признаков. Система не применяет изменения автоматически, ожидая подтверждения оператора.
249: Рисунок 7 – Экран обработки данных в пользовательском интерфейсе.
250: Экран настройки критериев поддерживает выбор целевого объекта, ручную корректировку весов и автоматическую нормализацию их суммы (Рисунок 8).
252: Рисунок 8 – Экран настройки критериев сравнения в пользовательском интерфейсе.
253: Визуализация результатов построена вокруг горизонтальной диаграммы рейтинга, дополненной таблицами с пагинацией, показателями надежности и перечнем аналогов с пояснением вклада каждого критерия (Рисунок 9).
255: Рисунок 9 – Экран просмотра результата в пользовательском интерфейсе.
256: Особое внимание уделено устойчивости работы: при загрузке крупных наборов данных интерфейс сохраняет отзывчивость за счет асинхронного взаимодействия с сервером и чанковой отдачи предпросмотра.
257: Навигационная логика исключает переход на следующий этап до завершения валидации текущего. Ошибки валидации выводятся в человеческом формате с указанием конкретного поля и причины отказа. Модальные окна заменены на встроенные блоки уведомлений, что снижает когнитивную нагрузку и предотвращает потерю контекста работы. История сравнений и экран проектов вынесены в отдельные разделы, доступные после авторизации, что позволяет сравнивать результаты различных сценариев и возвращаться к предыдущим расчетам.
